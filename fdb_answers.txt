Q1:
FoundationDB separates TMS and SS to isolate coordination from storage. TMS handles ordering, conflicts, and versions. SS stores data and serves reads, enabling scalability, fault tolerance, and independent layer optimization.

Q2:
The Proxy detects read-write conflicts during commit by checking the transaction’s read set against recent writes before assigning a commit version, ensuring serializability.

Q3:
The read version provides a consistent snapshot assigned by the Sequencer at transaction start, making sure all reads see a stable state and enforcing strong consistency guarantees.

Q4:
Snapshot reads skip read conflict tracking, avoiding conflicts with concurrent writes. They improve performance when slightly stale or non-critical data is acceptable.

Q5:
Recovery is cheap because LogServers durably store updates first, and StorageServers asynchronously apply them. Recovery uses logs directly without replaying a redo log in the critical path.
